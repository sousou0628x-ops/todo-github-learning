GitHub学習プロジェクト - 実践的教育計画
=====================================

【コンセプト】
個人開発者が実際に遭遇する失敗ケースを体験し、
GitHubがどう問題を解決するかを実感する

=====================================
GitHubのメリット・デメリット・なぜ使うのか
=====================================

【個人開発でよくある失敗ケース】

❌ 失敗1: 「動いてたコードが壊れた！でも戻せない」
   → ファイルを直接編集して、どこを変えたか忘れた
   → 「backup_最終版_本当に最終.js」みたいなファイルが増える
   ✅ GitHubの解決策: コミット履歴で全ての変更を追跡、いつでも戻せる

❌ 失敗2: 「新機能を試したいけど、今の動くコードを壊したくない」
   → コードをコメントアウトして、ぐちゃぐちゃになる
   → 実験的なコードと本番コードが混ざる
   ✅ GitHubの解決策: ブランチで安全に実験、失敗したら捨てるだけ

❌ 失敗3: 「PCが壊れた！コードが全部消えた」
   → ローカルにしか保存してない
   → バックアップを取り忘れる
   ✅ GitHubの解決策: クラウドに自動バックアップ、どこからでもアクセス

❌ 失敗4: 「3ヶ月前の自分が何を考えてこのコード書いたか分からない」
   → コメントもメモもない
   → なぜこの実装にしたのか思い出せない
   ✅ GitHubの解決策: コミットメッセージで意図を記録、Issue で議論を保存

❌ 失敗5: 「複数の機能を同時に開発してたら、どれがどれか分からなくなった」
   → 全部まとめて変更してしまう
   → バグの原因が特定できない
   ✅ GitHubの解決策: ブランチとPull Requestで変更を整理

【GitHubのメリット】
✅ タイムマシン機能（いつでも過去に戻れる）
✅ 並行世界で実験できる（ブランチ）
✅ 自動バックアップ（クラウド保存）
✅ 変更の理由を記録できる（コミットメッセージ、Issue）
✅ 自動テスト・自動デプロイ（GitHub Actions）
✅ ポートフォリオになる（就職・転職で見せられる）
✅ オープンソースに貢献できる

【GitHubのデメリット】
❌ 学習コストがある（最初は難しい）
❌ 小さなプロジェクトには過剰に感じる
❌ プライベートリポジトリでも、GitHubのサーバーにコードがある
❌ コマンドを間違えると混乱する
❌ 一人開発だと恩恵を感じにくい（でも後で役立つ）

【なぜみんな使うのか】
→ 上記の失敗を全て経験済みだから
→ チーム開発では必須（業界標準）
→ 一度慣れると、使わない方が怖い

=====================================
ステップ1: 「動いてたコードが壊れた」を体験→解決
=====================================

【失敗シナリオを再現】
あなたは今、動くToDoアプリを持っています。
「ちょっとデザインを変えよう」と思って、CSSをいじります。
でも、変更しすぎて元に戻せなくなります...

【学習目標】
- コミット履歴の価値を実感
- git log, git diff の使い方
- git checkout で過去に戻る方法
- 「変更を記録する」習慣の重要性

【実践演習】
1. わざとCSSを壊してみる
   - style.cssを大幅に変更
   - 「あれ？元のデザインの方が良かった...」

2. Gitで救出
   - git log で履歴を確認
   - git diff で何を変えたか確認
   - git checkout で元に戻す

3. 正しい変更の仕方
   - 小さな変更ごとにコミット
   - 意味のあるコミットメッセージ
   - 「いつでも戻れる」安心感

【体験する失敗】
「backup.css」「style_old.css」「style_最終版.css」
みたいなファイルを作る悪習慣

【GitHubの強み】
→ コミット履歴が自動的にバックアップになる
→ ファイルをコピーする必要がない
→ 変更の理由も一緒に記録できる

成果物：
- コミット履歴の重要性を体感
- git log, git diff, git checkout のスキル

=====================================
ステップ2: 「新機能を試したいけど壊したくない」を体験→解決
=====================================

【失敗シナリオを再現】
「ダークモード機能を追加したい！」
でも、今の動くコードを壊したくない...
→ コードをコメントアウトしまくる
→ if (false) { /* 古いコード */ } みたいなコードが増える
→ どれが本番用か分からなくなる

【学習目標】
- ブランチの価値を実感
- 「安全に実験できる場所」の重要性
- 失敗を恐れずに挑戦できる環境
- マージとコンフリクト解決

【実践演習】
1. ブランチなしで開発（失敗体験）
   - mainブランチで直接ダークモード機能を追加
   - 「やっぱりやめたい」と思っても戻せない
   - コードがぐちゃぐちゃに

2. ブランチで開発（正しい方法）
   - feature/dark-mode ブランチを作成
   - 自由に実験
   - 気に入らなければブランチごと削除
   - 気に入ったらmainにマージ

3. 複数の実験を並行
   - feature/dark-mode
   - feature/priority-tags
   - feature/due-date
   → それぞれ独立して開発、好きなものだけマージ

【体験する失敗】
コメントアウトだらけのコード：
```javascript
// 古い実装
// function addTask() { ... }

// 新しい実装（テスト中）
function addTask() { ... }

// さらに新しい実装（こっちの方がいい？）
// function addTask() { ... }
```

【GitHubの強み】
→ ブランチで並行世界を作れる
→ 失敗しても本番コードは無傷
→ 複数の実験を同時進行できる
→ チーム開発でも衝突しない

成果物：
- ブランチ戦略の理解
- feature ブランチでの開発経験
- マージの成功体験
- 「安心して実験できる」感覚

=====================================
ステップ3: 「なぜこのコードを書いたか忘れた」を体験→解決
=====================================

【失敗シナリオを再現】
3ヶ月後、自分のコードを見返す：
「なんでこんな複雑な実装にしたんだ？」
「このバグ修正、何のバグだったっけ？」
「この機能、使ってる人いるのかな？」
→ 記録がないので、思い出せない
→ 怖くて変更できない

【学習目標】
- Issue で「なぜ」を記録する重要性
- Pull Request で変更の文脈を残す
- 未来の自分へのドキュメント
- プロジェクト管理の基礎

【実践演習】
1. 記録なしで開発（失敗体験）
   - コミットメッセージ: "fix"
   - なぜ修正したのか不明
   - 後で見ても意味不明

2. Issue駆動開発（正しい方法）
   - Issue #1: 「タスク削除時に確認ダイアログがない」
     - 問題の説明
     - 再現手順
     - 期待する動作
   
   - Issue #2: 「完了タスクが多すぎて見づらい」
     - スクリーンショット添付
     - ユーザーの声
     - 解決案の議論

3. Pull Request で変更を説明
   - 「Issue #1 を解決: 削除確認ダイアログを追加」
   - 変更内容の説明
   - スクリーンショット（Before/After）
   - テスト方法

【体験する失敗】
意味不明なコミットメッセージ：
- "update"
- "fix bug"
- "change"
- "test"
→ 3ヶ月後、何も思い出せない

【GitHubの強み】
→ Issue で「なぜ」を記録
→ Pull Request で「何を」「どう」変えたか記録
→ 議論の履歴が残る
→ 未来の自分（や他の人）が理解できる
→ プロジェクトの歴史が可視化される

成果物：
- 意味のある Issue の作成
- 詳細な Pull Request
- Issue/PR テンプレート
- プロジェクトボードでの管理
- 「記録する」習慣

=====================================
ステップ4: 「手動デプロイが面倒」「バグに気づかない」を体験→解決
=====================================

【失敗シナリオを再現】
毎回手動でデプロイ：
1. コードを変更
2. FTPでアップロード
3. 「あれ？動かない...」
4. ブラウザのキャッシュクリア
5. 「まだ動かない...」
6. 「あ、ファイル上げ忘れた」
7. もう一度アップロード
→ 時間がかかる、ミスが多い、テストを忘れる

【学習目標】
- GitHub Actions で自動化の価値を実感
- 「プッシュしたら自動でデプロイ」の快適さ
- 自動テストでバグを早期発見
- CI/CD の基礎

【実践演習】
1. 手動デプロイの苦労（失敗体験）
   - ファイルを一つずつアップロード
   - 「どのファイルを変更したっけ？」
   - アップロード忘れでバグ発生

2. GitHub Actions で自動化（正しい方法）
   - .github/workflows/deploy.yml を作成
   - main ブランチにプッシュ → 自動デプロイ
   - GitHub Pages で公開

3. 自動テストの追加
   - HTMLバリデーション
   - JavaScriptの構文チェック
   - 「壊れたコードはマージできない」仕組み

4. 自動化の恩恵を実感
   - git push するだけで完了
   - テストが自動で走る
   - デプロイも自動
   - 「何もしなくていい」快適さ

【体験する失敗】
手動デプロイのミス：
- ファイルのアップロード忘れ
- 古いファイルが残ってる
- 本番環境で直接編集してしまう
- バックアップを取り忘れる
- 「動いてたのに壊れた」

【GitHubの強み】
→ プッシュするだけで自動デプロイ
→ テストが自動で走る
→ 壊れたコードは本番に行かない
→ デプロイ履歴が残る
→ ロールバックも簡単
→ 「人間がやらなくていい」ことは自動化

成果物：
- GitHub Actions ワークフロー
- GitHub Pages での公開
- 自動テストの設定
- 「自動化の快適さ」の体験
- CI/CD の理解

2. GitHubの実践的スキル

次のステップ

   - 「プロはこう書くのか」を学ぶ
   - Issue を読んで議論を見る
   - 小さなバグ修正から貢献

2. 自分のプロジェクトをGitHubで管理

   - Git rebase
   - Git cherry-pick
   - GitHub Actions の高度な使い方
   - セキュリティ機能（Dependabot等）

=====================================
重要な心構え
=====================================

「GitHubは難しい」と感じるのは普通です。
でも、一度慣れると「GitHubなしでは開発できない」と感じるようになります。

失敗を恐れず、たくさん実験してください。
Gitがあれば、いつでもやり直せます。

この学習プロジェクトで、
「GitHubって便利だな」と実感してもらえたら成功です。
